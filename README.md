# 💼 Loan Intelligence Portal (Machine Learning End-to-End)

Welcome to the **Loan Underwriting and Intelligence Portal**, an end-to-end Machine Learning solution designed to predict loan application eligibility in real-time. This project handles the entire data lifecycle: from raw data exploration and cleaning, feature engineering, and model optimization to local deployment as an interactive dashboard.

---

## 🎯 What the Application Does
The **Loan Underwriting and Intelligence Portal** is a production-grade web application built using **Streamlit** that acts as an automated underwriting engine. It bridges the gap between machine learning models and end-users (such as credit analysts or loan officers) by providing a premium, interactive interface to evaluate loan applications.

Key capabilities of the application:
1. **Interactive Profiling:** Allows users to input applicant profile details (Dependents, Credit History) and financial fields (Applicant Income, Coapplicant Income, Requested Loan Amount, Loan Term).
2. **On-the-Fly Pipeline Alignment:** Automatically cleans, handles missing inputs, and applies the exact preprocessing statistics derived from the training dataset (such as imputing missing values with training medians/modes and capping values with Interquartile Range outlier limits) to prevent data leakage.
3. **Real-Time Feature Engineering:** Dynamically generates complex ratios (Debt-to-Income, Total Income, Loan-to-Income) and ordinal quantile bins for total income and loan amounts before feeding them to the predictor.
4. **Automated Evaluation:** Underwrites the application in real-time using a serialized, highly optimized **XGBoost Classifier** model and displays a clear approval or denial recommendation with model confidence scoring.
5. **Pipeline Traceability:** Displays a complete dashboard containing the exact numerical vector generated by the pipeline and fed to the model, ensuring transparency and reproducibility.

## ⚙️ Preprocessing & Feature Engineering Pipeline

During inference, user input is processed using the exact same metrics and bounds calculated from the training dataset to prevent data leakage and guarantee prediction alignment:

1. **Imputation:** Missing numerical values are filled using training medians (e.g., `ApplicantIncome` median: `$3,812.50`, `LoanAmount` median: `$128,000`).
2. **Outlier Capping:** Input values are clipped using the training set's Interquartile Range (IQR) limits to protect the model from extreme inputs:
   * `ApplicantIncome` is clipped to `[-1,498.75, 10,171.25]`
   * `CoapplicantIncome` is clipped to `[-3,445.88, 5,743.13]`
   * `LoanAmount` is clipped to `[3.50, 261.50]`
3. **Feature Engineering:**
   * **Debt-to-Income (DTI) Ratio:** $\text{LoanAmount} / (\text{ApplicantIncome} + 1)$
   * **Total Income:** $\text{ApplicantIncome} + \text{CoapplicantIncome}$
   * **Loan-to-Income Ratio:** $\text{LoanAmount} / (\text{ApplicantIncome} + 1)$
   * **Quantile Binning:** Total Income and Loan Amount are categorized into 4 ordinal groups based on the training quantiles and encoded to codes `0`, `1`, `2`, or `3`.
4. **Categorical Alignment:** Categorical features that were coerced to numeric codes or 0 in the notebook's training code (e.g., `Gender`, `Married`, `Education`, `Self_Employed`, `Property_Area`) are kept in the feature list as constant markers to maintain model compatibility without introducing noise.

## 🧠 Algorithms Comparison & Implementation

Three classification models were trained and tuned using cross-validation to select the most reliable underwriter. The table below outlines how each algorithm works and how it was implemented:

| Algorithm | Theoretical Concept & Purpose | Implementation Details | Preprocessing & Pipeline Alignment |
| :--- | :--- | :--- | :--- |
| **XGBoost Classifier** | An optimized gradient boosting framework. It builds weak decision trees sequentially, with each new tree correcting the residual errors of the previous ones using gradient descent. It handles complex, non-linear relationships and feature interactions extremely well. | - Class `xgboost.XGBClassifier` was tuned via `GridSearchCV` (3-fold CV). <br> - Imbalance was addressed by computing `scale_pos_weight` as the ratio of non-defaults to defaults (~2.0). <br> - Grid parameters searched: `max_depth` (3, 5, 7) and `learning_rate` (0.01, 0.1, 0.2). | - Quantile binning for income and loan amounts. <br> - Outlier capping with IQR. <br> - Natural handling of non-linear markers; no feature scaling required. |
| **Random Forest Classifier** | An ensemble bagging method that builds multiple independent decision trees on random subsets of training data (bootstrap samples) and features. The final prediction is determined by a majority vote of all trees, reducing variance and overfitting. | - Class `sklearn.ensemble.RandomForestClassifier` was tuned via `GridSearchCV` (5-fold CV). <br> - Class imbalance handled using `class_weight='balanced'`. <br> - Grid parameters searched: `n_estimators` (100, 200, 300), `max_depth` (None, 5, 10, 15), and `min_samples_split` (2, 5, 10). | - Uses the same preprocessed and cleaned datasets. <br> - Tree-based architecture does not require normalized scale features. |
| **Logistic Regression** | A baseline statistical model that estimates the probability of a binary outcome using a logistic sigmoid function. It models the decision boundary as a linear combination of input features. | - Class `sklearn.linear_model.LogisticRegression` was wrapped in an estimator `Pipeline` containing scaling logic. <br> - Class imbalance handled using `class_weight='balanced'`. <br> - Grid parameters searched: penalty (`l1`, `l2`), regularisation strength `C` (0.01, 0.1, 1, 10, 100), and solvers (`liblinear`, `saga`). | - **Crucially requires feature scaling** (using `StandardScaler` inside the pipeline) because distance-based gradient updates are highly sensitive to feature scales. |

## 📈 Model Performance & Result Explanations

### Comparative Metrics (on Validation/Test Set)
Based on the grid search and comparative evaluation in the notebooks:

| Model | Test Accuracy | F1 Score | ROC-AUC |
| :--- | :---: | :---: | :---: |
| **XGBoost (Best Tuned)** | **77.24%** | **85.57%** | **0.8427** |
| **Random Forest** | 69.11% | 78.89% | 0.5774 |
| **Logistic Regression** | 56.10% | 63.01% | 0.5579 |

---

### Detailed Analysis and Explanations

#### 1. Why XGBoost Performed Best (Winner)
* **Capturing Non-Linear relationships:** Credit scoring data contains highly complex and non-linear interactions. For example, a high loan amount is not necessarily bad, but it becomes critical when combined with low applicant income and bad credit history. XGBoost's sequential boosting nature allows it to continuously target and split on these subtle combinations of features.
* **Handling Imbalance:** By incorporating `scale_pos_weight` directly in the training objective, the algorithm penalized false negatives heavily, achieving a strong balance between precision and recall, resulting in a superior **F1 Score of 85.57%** and **ROC-AUC of 0.8427**.
* **Robustness to Preprocessing:** XGBoost natively ignores absolute feature scale differences and easily handles features transformed via clipping and quantile binning.

#### 2. Why Random Forest Achieved Moderate Performance
* **Bagging vs. Boosting:** Random Forest achieved **69.11% Accuracy** and **78.89% F1 Score**. It is an ensemble model that builds independent trees. While it successfully handles non-linear variables, it lacks XGBoost's boosting mechanism, which focuses on "hard" examples. 
* **Sensitivity to Noise:** In smaller datasets (such as this loan dataset with around 614 samples), random forest trees can overfit to random noise in deep nodes, leading to a much lower **ROC-AUC (0.5774)** compared to XGBoost.

#### 3. Why Logistic Regression Performed Poorest
* **Linear Assumption Constraint:** Logistic Regression struggled significantly, yielding **56.10% Accuracy** and **0.5579 ROC-AUC**. It assumes that the boundary separating approved and denied loans is a straight hyper-plane. In reality, credit risk decision boundaries are highly non-linear.
* **Lack of Feature Interaction Capability:** Logistic Regression cannot model feature interactions (e.g., the relationship between income and loan term) unless they are manually generated (e.g., polynomial features). Without extensive manual feature engineering, it suffers from high bias (underfitting).
